## What I built

### libs/core/src/mission.ts (~190 lines)

YAML-frontmatter + activity-log parser. Strategy:

1. `splitFrontmatter(source)` — regex splits `---\n...\n---\n<body>`; YAML-parse the middle. Tolerates files with no frontmatter (returns empty `{}`) + malformed YAML (returns `null` frontmatter, parses body anyway).
2. `parseActivityLog(body)` — walks lines for `## Activity log`, then each subsequent `- <ISO>  <agent>  [session=<uuid>  ]<text>` bullet line. Stops at next `## heading`. Regex-extracts the C8 `session=<uuid>` token if present and strips it from `text`.
3. Field coercion helpers — `coerceStatus` / `coerceStringArray` / `coerceNumOrNull` / `coerceStringOrNull` / `coerceTokens`. All permissive, no throws.
4. **Mission-ID leading-zero bug fix** — YAML parses `mission: 0029` as number `29`. Detect numeric input and `String(n).padStart(4, "0")` to restore "0029".
5. `findMissionFolder(workspacePath, id)` — lists `<ws>/missions/` and returns the first entry starting with `<id>-`.

Three exports added to `@team-orchestration/core`: `parseMissionState`, `readMissionStateFile`, `findMissionFolder`.

## Decisions I made

- **`yaml` over `js-yaml`.** Smaller (~30KB vs ~70KB), more actively maintained in 2024–26, native ESM. Drop-in for our use case.
- **Permissive parser.** Empty frontmatter → empty `MissionState` with empty activity log. Never throw. Dashboard degrades gracefully — if a mission's state.md is malformed, the drill-down view shows what's parseable + marks gaps.
- **Leading-zero mission-id coercion in `mission.ts`, not `shared-types`.** The type is `string`; the parser is responsible for producing it from whatever YAML returns.
- **Default `--log-limit 10`, not unlimited.** Active missions have 20+ activity-log entries. Showing the last 10 reads well in a terminal; user bypasses with `--log-limit 0`.
- **Session-tag badge shows first 8 chars of UUID**, not full. Full UUIDs are unreadable in a table; 8 chars is enough to distinguish and disambiguate.
- **140-char truncation on activity text** — enough for most supervisor notes to convey intent; long "wall of text" entries get truncated with `…` which signals there's more. Future: `--full` flag to disable truncation.
- **Gate history + Pause history are NOT parsed yet.** Section-boundary regex stops at `## ` which means subsequent sections are ignored. Rendering them is a wave-3 concern when the UX decision is made ("inline in drill-down" vs "separate `--show-gates` flag").
