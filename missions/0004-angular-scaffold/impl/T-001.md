## What I built

### apps/dashboard-server (Node stdlib HTTP, ~200 lines)

Zero external deps — `createServer` + `URL` from `node:http` + `node:url`. Five routes:

```
GET  /api/health                        → { ok, version, timestamp }
GET  /api/workspaces                    → WorkspaceInfo[]
GET  /api/missions?path=<ws>            → MissionRegistryRow[]
GET  /api/mission?path=<ws>&id=<NNNN>   → MissionState
GET  /api/telemetry?path=<ws>&id=<NNNN> → SessionSlice
```

CORS preflight handled. Port default 4117 (env `TOT_DASHBOARD_PORT` overridable). All route handlers `await` on `@team-orchestration/core` functions — this server is just a transport; logic lives in the lib.

**Architecture mirror:** this HTTP contract is exactly what Electron's IPC will expose in wave 5. Swap `fetch('/api/…')` for `window.api.…()` in `ApiService` and the Angular app runs inside Electron unchanged. The 3 endpoints that take `path=` are how the dashboard asks the "backend" to operate on a specific workspace.

## Decisions I made

- **Hand-scaffold Angular, don't use Nx generator.** Nx generators write many files + may not match our layout preferences (standalone-only, zoneless, custom folder shape). Hand-writing `angular.json` + minimal `tsconfig.app.json` gets us the exact stack we want in ~30 lines of config. Nx-level orchestration still available via pnpm workspaces; we can add Nx `project.json` files later if Nx Cloud caching becomes valuable.
- **Node stdlib HTTP, not Fastify/Hono.** 3 routes + CORS is ~50 lines of stdlib — fewer than even Hono's config. Zero external deps keeps the server boot < 100ms. The next wave adds Electron where stdlib also works.
- **Hash router (`withHashLocation`)** — works in `file://` / Electron packaged builds without server-side rewrites. Trade-off: URLs have `#` in them; acceptable for a local dashboard.
- **Zoneless change detection.** All state goes through signals. No `Zone.js` in the bundle → smaller, faster. Consistent with modern Angular philosophy.
- **`@angular/build` over `@angular-devkit/build-angular` webpack.** esbuild is the Angular 21 default; 3× faster builds, smaller config. Both listed as devDeps because some older Nx integrations still reach for `@angular-devkit/build-angular`; doesn't cost us anything to ship both.
- **Two apps, not a monolith.** Server + renderer are *physically* split because in wave 5, "server" becomes "Electron main process" (different Node context). Forcing the split now means wave 5 is a "swap transport, not restructure" mission.
- **`ApiService` is the only seam against the transport.** Components never call `fetch` or `HttpClient` directly. In wave 5, `ApiService` swaps from HttpClient → IPC in one file change.
- **Bumped TS to 5.9** rather than pinning Angular to an older version. Angular 21 is the user's stated target; compiling against an outdated TS defeats the choice.
- **No mission drill-down yet** — deferred to wave 4.5. User can click workspace → see missions list; drilling into one mission uses the CLI (`tot mission`) for now. Angular drill-down is pure presentation work — ~200 lines of Angular in a fresh component, no new library code. Separate commit keeps wave 4 reviewable.
