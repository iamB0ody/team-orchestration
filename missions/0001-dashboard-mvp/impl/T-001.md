## What I built

### libs/core/src/registry.ts (~200 lines)
Markdown parser for `missions/REGISTRY.md`. Strategy:
1. Walk lines; track current H2 section (`## Active` / `## Done` / `## Cancelled` / `## Archived`).
2. Skip the header row (first cell is "ID") + the `|---|---|` separator row.
3. Per row: split on `|`, strip padding, map cells to the table's schema.
4. Permissive parsing — legacy `feature *(legacy schema)*` → `feature`; empty / dash cells → `null`; backtick SHAs stripped; dep lists like `[0003, 0005]` → `["0003", "0005"]`.

Two exports:
- `parseRegistry(markdown: string): MissionRegistryRow[]` — pure.
- `readRegistryFile(path: string): Promise<MissionRegistryRow[]>` — convenience.

### libs/core/src/workspace.ts (~80 lines)
Reads `$XDG_CONFIG_HOME/team-orchestration/workspaces.json` (falls back to `~/.config/team-orchestration/workspaces.json`). For each registered workspace, runs the registry parser against its `missions/REGISTRY.md` and returns a `WorkspaceInfo[]` with active/done counts + last-update date.

## Decisions I made

- **CLI-first, not Angular-first.** The fastest path to validating the parser is a CLI. Also gives the user immediate value — a working `tot` command on day one, not "wait weeks for the Electron app." The Angular+Electron shell wraps this same parser lib in later waves.
- **tsx for dev runtime.** No compile step during development. Keeps the loop fast. Production packaging (for `pnpm link -g tot`) is a later concern.
- **picocolors over chalk.** picocolors is 13 lines + zero deps. Chalk adds ~30KB for no feature we need.
- **XDG config path.** `$XDG_CONFIG_HOME/team-orchestration/workspaces.json` with `~/.config/...` fallback. Cross-platform-friendly + matches Linux convention + users already know it from most dev tools.
- **Permissive parser.** Legacy Shamil rows (e.g. `feature *(legacy schema)*`) must parse. Unknown cells → `null`, never throw. Future-proofs against SKILL.md schema evolution.
- **Zero Angular/Electron code this wave.** libs/ui/tokens/theme.scss is already there from mission 0034 — waiting for components. libs/ipc is empty — waiting for the Electron main process. Shipping small, shipping often.
- **No state.md / transcript / join code yet.** Those are their own concerns (mission.ts, transcript.ts, join.ts) and each warrants its own mission with proper testing. Registry parsing alone is enough to render the top-level missions list; state.md details come when the user drills into a row.
- **No CTO retro auto-fire.** The arc has been heavy on meta-loops (0023 / 0027 / 0028 / 0029 / 0034 all had retros). Wave 1 of a fresh FEATURE-S inside a new repo doesn't need one. User triggers on request.
